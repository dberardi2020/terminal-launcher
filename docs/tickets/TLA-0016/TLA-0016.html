TLA-0016 — Non-interactive config CLI

The config-editing verbs (new, edit, pane-new) are prompt-driven: they walk you through a wizard and write back. There is no way to create or edit a pane or workspace in one non-interactive call, so anything scripted has to either drive the prompts or hand-edit workspaces.json and hope it stays valid.

Why it matters

  • Scripting and reproducible setup — seeding a machine, or a dotfiles repo, currently means shipping a whole config file rather than a few commands.
  • CI — no way to build a config fixture without the wizard.
  • Agent-driven setup — the README's "hand it to your coding agent" flow stops at init; an agent can't compose a workspace for you without simulating keystrokes.
  • Demo/test fixtures — the demo screenshots needed a hand-written config (see TLA-0017).

Proposed surface

Flag-driven variants alongside the existing interactive ones:

terminal-launcher pane-new --name API --color green --target ~/src/api --model opus
terminal-launcher workspace set "Full Stack" --layout quad --slots api,web,infra,tests
terminal-launcher workspace set Backend --layout split --slots api,infra --flip

Plus a stdin-JSON form for bulk/idempotent use:

echo '{"panes": {...}, "workspaces": [...]}' | terminal-launcher import --merge

Design notes

  • Thin wrapper, not a second engine. The config is already a single JSON file with config.load / config.save (atomic) and model.resolve_workspace for validation. These verbs should reuse that path so the interactive and non-interactive routes cannot diverge — the same property the CLI and GUI already have.
  • Validate before writing. Reuse CompositionError (unknown layout, dangling pane ref) so a bad flag combination fails loudly rather than persisting a broken config.
  • Slot model precedence. A slot stores model only when it overrides the pane default; the non-interactive form must preserve that, not write the resolved value.
  • Decide whether set is create-or-update (idempotent, good for scripts) or whether create and update stay separate verbs.

Open questions

  • Does this replace the wizard for edit, or sit beside it?
  • Is import --merge worth it, or is per-entity flags enough?
