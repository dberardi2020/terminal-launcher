# TLA-0023 — Audit error handling, reporting & logging

Both halves sweep the *same* call sites — `_paste_command`, `cmd_restore`,
`backend.launch`, and the GUI bridge all log **and** report — so they stay one ticket.
Split if (a) alone proves large enough.

## Why now

Two "failure looked like success" defects turned up in a single feature (`/restore`),
both in code that read fine:

- **Windows** — `_paste_command` only *logged* a miss while `cmd_restore` printed
  `RESTORED` regardless, so the slash command told the user *"Restored `<name>`"* for a
  paste that never happened.
- **macOS** — `restore_current` let exceptions escape into the iterm2 runner, which
  catches-and-prints and exits 1, bypassing the `ERROR` branch entirely.

Both are fixed. The point of the audit is that neither was *visible* by reading the
call site — the reporting contract lived in one module and the failure in another.

## (a) Errors

Sweep every failure path and ask: does it propagate, is failure distinguishable from
success, is the message actionable, is the documented contract honoured
(`DETECTED` / `RESTORED` / `UNKNOWN` / `ERROR` + exit codes)?

Specific areas:

- **`backend.launch`** — per-slot failure partway through a multi-pane launch.
- **The pywebview bridge** — pywebview turns exceptions into silently-rejected JS
  promises; that's why `gui.Api` wraps every public method in a tracer. Confirm the
  tracer's coverage is complete.
- **CLI exit codes** — audit against the documented contract in `cmd_restore`'s
  docstring.

### Write down the lenient-vs-strict convention

This is the load-bearing decision, currently implicit in the code:

- **Launch-time `/color` is lenient** — `_inject_color` ignores `_paste_command`'s
  return. One pane failing to colour must not abort a quad launch.
- **`restore` is strict** — `restore_current` raises, so a failed injection reaches the
  CLI's `ERROR` branch rather than being reported as success.

Both are correct, but nothing states the rule, so the next call site is a coin flip.

## (b) Logging

### Known: the stderr banner

`diag.setup()` attaches a `StreamHandler` alongside the rotating file handler, so
`INFO: --- diag ready (log at …) ---` prints on **every** CLI invocation before its real
output. Observed on every run during the Windows verification.

It's on stderr, so it can't corrupt `restore`'s tab-delimited stdout contract — but it
*is* visible in the `/restore` slash-command output the user reads. Options: drop the
banner to `DEBUG`, or attach the stderr handler only for the GUI / long-running paths.

### Known: no verbosity control

Level is hardcoded `DEBUG` on the logger, with no `--verbose` / `--quiet` flag, no env
var, and no config key. Both handlers inherit it.

### Known: double-reporting

Raised failures shouldn't *also* log a warning. Note the Windows `/restore` fix
**introduced** an instance of this: `_paste_command` logs a warning on a focus failure
*and* returns `False`, on which `restore_current` raises. One event, two reports.

### Found during the Windows verification — `restore.py` has no logging at all

`restore.py` is the newest surface and has **zero** log calls. Nothing records:

- which pane targets were considered against the cwd,
- which one won a longest-match tie,
- whether the per-session sentinel was read, written, or missed.

"Why did it pick that pane?" is this feature's primary failure mode, and today the only
way to answer it is to re-run with `--detect-only` and infer. The backends already log
this way (`_log.info("wt launch: layout=%s …")`); `detect()` should too.

Note the ticket's existing wording covers *pane-launch* failures logging enough context —
detection is a separate path and is not covered by that.

### Found during the Windows verification — no run attribution

The log format carries no PID or run-id:

```
2026-07-21 12:34:29 INFO  [terminal_launcher] wt launch: layout=single flip=False slots=1 filled=1
```

Concurrency is expected — the wt backend holds a named mutex
(`TerminalLauncher.ColorInjection`) precisely so simultaneous launches don't clobber each
other's clipboard and foreground. When two interleave, their lines can't be told apart.

Also note `[%(name)s]` is always the literal `terminal_launcher`: `diag.get_logger()`
returns one shared logger rather than per-module `__name__` loggers, so the field carries
no information and the module a line came from is unrecoverable.

## Current state, for reference

| | |
|---|---|
| Sink | `~/.config/terminal-launcher/terminal-launcher.log`, `RotatingFileHandler` 512 KB × 2, UTF-8 |
| Second sink | `StreamHandler` → stderr |
| Level | `DEBUG`, hardcoded |
| Format | `%(asctime)s %(levelname)-5s [%(name)s] %(message)s` |
| Uncaught | `sys.excepthook` logs `UNCAUGHT` with traceback, then delegates |
| Read paths | `logs` CLI verb (`read_tail`, 200 lines); GUI `get_diagnostics` (300), `export_logs`, reveal-log button |
| `setup()` callers | `cli.py`, `gui.py` — idempotent via a module-level `_configured` |

Call sites by module — the backends log, and effectively nothing else does:

| Module | Calls |
|---|---|
| `iterm2_backend.py` | 9 |
| `windows_terminal_backend.py` | 6 |
| `gui.py` | 6 |
| `cli.py`, `backend.py`, `config.py`, `restore.py`, `model.py`, `layouts.py` | 0 |

Worth keeping through any refactor: WebView JS errors (`window.onerror`,
`unhandledrejection`, `console.error`) are forwarded over the pywebview bridge via
`Api.log_client` into the same file, so a composer bug leaves a record instead of
vanishing into a WebView console nobody can open.
