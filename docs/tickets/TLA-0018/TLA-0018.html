TLA-0018 — Hosted browser demo (GitHub Pages)

A "try it in your browser" page for the visual composer, linkable from the README.

Why it's feasible

The composer is already a single self-contained HTML file (terminal_launcher/web/builder.html — inline CSS/JS, no external assets) talking to Python through one narrow seam:

const api = () => window.pywebview.api;

Every backend call goes through that accessor. Stub window.pywebview.api with an in-memory JavaScript mock — a canned get_state(), and save_workspace / save_pane / launch as no-ops against a local object — and the same file runs as a static page with no Python at all. That's a GitHub Pages deploy, not a rewrite.

What it would and wouldn't be

  • Would: the real UI, real layouts, real interactions — click a card, load it into the composer, assign panes, switch layouts, open the pane manager.
  • Wouldn't: launch anything. A browser can't spawn terminals, so Launch has to be a no-op or a toast. It is a UI tour, not a functional demo, and the page should say so plainly rather than let someone conclude the tool is broken.

Costs to weigh

  • The mock is a second implementation of the API contract and will drift from gui.Api unless something keeps them honest. That is the real long-term cost, not the initial build.
  • Needs a demo config anyway — the same fixture problem as TLA-0017.

Verdict

Nice-to-have, not blocking. The three README screenshots already answer "what does it look like?" at a fraction of the cost. Worth revisiting if the project attracts enough interest that "let me try it" beats "let me look at it".
